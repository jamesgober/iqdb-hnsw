# iqdb-hnsw v0.6.0 — Feature-complete, recall-validated, API-frozen

**The graph is built and the surface is locked.** v0.1.0 was the scaffold; v0.6.0
lands the whole HNSW index onto it, wires it to the stable (`1.0`) iQDB spine,
validates recall, and commits the public API for the 1.x series. The roadmap's
v0.2 (graph + insert + search), v0.3 (neighbour heuristic + tombstone), v0.4
(filtered traversal + feature freeze), and v0.5 (recall + API freeze) milestones
land together here. The release is numbered **0.6.0** to align with the iQDB
family's version line — `iqdb-flat` and `iqdb-build` are already at `0.5.0`+ — so
the crate goes straight from the scaffold to 0.6.0 (its alpha entry point).

## What is iqdb-hnsw?

The production approximate-nearest-neighbour layer of the iQDB vector database — a
Hierarchical Navigable Small World graph (Malkov–Yashunin, 2016) that answers
top-`k` queries by beam-search descent through a multi-layer proximity graph.
After the types crate it is the most important crate in the family: it is what
makes vector search over millions of vectors fast. It implements the
`iqdb-index` `Index` trait, so it is interchangeable with the exact `iqdb-flat`
and the clustered `iqdb-ivf` behind one interface.

## What's in 0.6.0

### The index

- **`HnswIndex`** implementing `iqdb_index::IndexCore` + `Index`
  (`type Config = HnswConfig`) — insert, delete (tombstone), batch insert, single
  and batch search, `flush`, and `stats`, plus inherent `new_unconfigured`,
  `dim`, `metric`, `len`, `is_empty`, `config`, and a `node_layer_histogram`
  diagnostic.
- **`HnswConfig`** — `m`, `ef_construction`, `ef_search`, `filter_widen`, `seed`,
  with builder-style `with_*` overrides and documented defaults (`m = 16`,
  `ef_construction = 200`, `ef_search = 64`, `filter_widen = 4`).

### Construction & search

- **Columnar graph storage** — `Vec<Arc<[f32]>>` row payloads plus parallel
  `Vec`s (ids, metadata, insertion-sequence, tombstone flags, per-node layer),
  per-node/per-layer adjacency, and an `id → node` map for `O(1)`
  duplicate/delete lookup.
- **INSERT-NODE (Alg 1)** with the Alg 4 diverse-neighbour heuristic,
  bidirectional linking, and overflow re-pruning; adjacency bounded by `m`
  (`2 * m` at layer 0).
- **Beam search** — greedy descent through the upper layers (`ef = 1`) then
  SEARCH-LAYER (Alg 2) at layer 0 with `ef = max(ef_search, k)`.

### Correctness guarantees

- **One ordering invariant.** All metric math goes through
  `iqdb_distance::compute_batch`; `DotProduct` is negated at the boundary so
  `Hit.distance` is *smaller-is-nearer* across all five metrics.
- **Deterministic.** The layer-assignment PRNG is a hand-rolled SplitMix64
  (no `rand` dependency) seeded from `HnswConfig::seed`. Identical insert order +
  seed ⇒ byte-identical graph and identical results, on every platform.
- **Tombstone delete.** A deleted node leaves traversal connectivity intact but
  is never returned as a hit; `len()` reports the live count.
- **Zero-copy insert.** The caller's `Arc<[f32]>` is stored verbatim.

### Filtered traversal

- `SearchParams::filter` is evaluated via `iqdb-filter`, with the beam widened by
  `filter_widen` (default `4×`) to mitigate post-filter under-return. A selective
  filter can still under-return; the limitation is documented on `search`.

## Recall validation

The headline gate (`tests/recall.rs`) asserts recall@10 ≥ 0.95 across **all five
metrics** on a 5,000-vector corpus, measured against an inline exact full-scan
oracle (the same `iqdb_distance` kernel, `DotProduct` negated) — so the test
needs no sibling index crate and measures the *graph*, not a second
implementation. An `#[ignore]`'d SIFT-1M recall diagnostic
(`tests/sift_recall.rs`) is available for real-data validation when the dataset
is present; an hnswlib comparison is deferred to the alpha phase as an external
benchmark.

## API freeze

The public surface is committed for the 1.x series; the full list is recorded in
`dev/ROADMAP.md`:

- `VERSION`, the `Hit` re-export, `HnswConfig`, and `HnswIndex`.
- `impl Index for HnswIndex` (`type Config = HnswConfig`) and the full
  `IndexCore` implementation.
- No feature flags.

Frozen behavioural contracts: smaller-is-nearer ordering across all five metrics
(`DotProduct` negated); deterministic graph under a fixed seed + insert order;
tombstone delete (a deleted id never reappears until re-inserted); no panics on
any input; zero `unsafe`. `iqdb_types::IqdbError` is `#[non_exhaustive]`, so new
error variants remain non-breaking. Only **additive, non-breaking** changes are
made from here through 1.x.

## Dependencies

Wired to the stable iQDB spine — all `1.0`:

```toml
iqdb-types    = "1.0.0"
iqdb-distance = "1.0.0"
iqdb-index    = "1.0.0"
iqdb-filter   = "1.0.0"
```

There is no other dependency, and no feature flags.

## Breaking changes

**None** in the API sense — v0.1.0 exposed no domain types. This release adds the
entire surface for the first time and freezes it.

## Verification

Clean across the CI matrix (Linux, macOS, Windows) on stable and the 1.87 MSRV:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo deny check
```

Zero `unsafe` (`#![forbid(unsafe_code)]`); no `unwrap`/`expect`/`todo!`/`dbg!` in
shipping code. The recall gate is also verified in release mode on Linux/x86
(exercising the AVX2 distance kernels).

## What's next

- **0.7.x → 0.9.x — alpha/beta/RC.** Integration against the engine (`iqdb`,
  `iqdb-build`), an hnswlib external comparison, real-data SIFT/GIST recall,
  broader testing, final benchmarks, and doc polish.
- **1.0.0 — stable**, once the Definition of Done holds and the API has proven
  itself against those consumers.
- **Slot compaction / graph-repair delete** is tracked as a post-freeze internal
  optimisation (see the ROADMAP note); delete is tombstone-only until then, and
  it does not change the public API.
